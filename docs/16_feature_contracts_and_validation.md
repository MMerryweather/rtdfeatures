# Feature Contracts and Validation

## Purpose

Explain how validation reduces risk without rebuilding MLOps.

## Core Principle

```
Documents explain.
Schemas constrain.
Validators block.
Tests prove.
MLOps stores.
```

Each layer has a distinct role. No single layer replaces the others.

## Feature Contracts Versus Passive Metadata

Feature contracts are machine-readable, validated, and enforced at execution time. Passive metadata is human-readable documentation that can go stale.

| Property | Feature contract | Passive metadata |
|---|---|---|
| machine-readable | yes | no |
| validated | yes | no |
| blocks invalid plans | yes | no |
| generated by code | yes | often hand-written |

`rtdfeatures` uses feature contracts (`FeatureSpec`, `KernelSpec`, `KernelFeaturePlan`) for execution. Passive metadata should be generated as evidence artifacts, not maintained by hand.

## Column Roles

Columns in a feature table carry implicit risk. `ColumnRole` is an optional machine-readable annotation for leakage and compatibility checks.

Examples:

- `input_tag` — process measurement used as a feature input
- `target` — the variable being predicted
- `lab_label` — infrequent lab assay used as ground truth
- `event` — operational event indicator
- `setpoint` — controller setpoint (may be future-known)
- `controller_output` — manipulated variable
- `downstream_measurement` — measurement that occurs after the target in process order
- `future_known` — known at feature time (e.g., scheduled feed blend)
- `forbidden` — must never appear as a feature

Column roles are advisory for `rtdfeatures` and enforced by `processfeaturecompiler.validation`.

## Observation Semantics

`ObservationSemantics` describes how a timestamp relates to the measurement:

- `instant` — measurement at a point in time
- `interval_start`, `interval_end`, `interval_midpoint` — interval-based measurements
- `online` — continuous analyser
- `grab` — spot sample
- `composite` — blended sample over an interval
- `event` — event-driven timestamp
- `derived` — computed from other observations

Semantics are optional in `rtdfeatures` but required for `processfeaturecompiler.validation` to detect horizon violations and temporal misalignment.

## Target Horizon

Feature plans must specify the target horizon: how far into the future the target extends relative to feature time. Standard options:

- `t` — concurrent prediction
- `t+1`, `t+k` — k-step ahead prediction
- `interval` — prediction over a future window

Horizon is part of `FeatureSpec` and validated by `FeaturePlanValidator`.

## Feature-Plan Validation

FeaturePlanValidator (`processfeaturecompiler.validation`) checks:

- column roles against feature usage
- target horizon against feature lookback
- no-future-data constraints
- graph-path validity
- recycle policy compliance
- feature-budget limits
- observation semantics compatibility

Validator output is a machine-readable `ValidationReport` with pass/fail/warn status per check.

## Leakage Checks `rtdfeatures` Can Enforce

- `ColumnRole.forbidden` columns are rejected as feature inputs
- `ColumnRole.future_known` is flagged when used without explicit acknowledgement
- temporal ordering is enforced during OOF execution (`fit_transform_oof`)
- `ForwardChainingSplitConfig` prevents future-data contamination in split generation

## Leakage Checks Left to Orchestration/MLOps

- split assignment integrity across feature and target tables
- timestamp alignment across data sources
- identical-future-data contamination from joined tables
- point-in-time correctness of external feature stores
- experiment-level leakage from repeated evaluation on correlated windows
- model-level leakage (feature selection on full data, etc.)

`rtdfeatures` enforces what it can at the kernel-feature level. Full pipeline leakage prevention requires orchestration.

## Feature Plan Manifests

`FeaturePlanManifest` is a reproducibility artifact produced after plan execution:

- plan hash
- schema hashes for input and output
- `rtdfeatures` package version
- warning summaries
- output schema summary

Manifests are machine-readable and designed to be stored by MLOps tooling alongside feature tables and model artifacts.

## Promotion Gates as External Policy

Feature promotion (staging to production) is an external policy decision. `rtdfeatures` and `processfeaturecompiler` provide the evidence and validation reports. The promotion gate is owned by MLOps or the team workflow.

## Synthetic Leakage-Trap Tests

Synthetic data with known leakage paths should be used to validate that validators catch violations. `rtdfeatures` synthetic data utilities (`synthetic.py`) support constructing leakage-trap scenarios for `v1.0` contracts.
